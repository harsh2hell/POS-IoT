# Security Policy

Thank you for helping keep this project and its users safe.  
This document outlines how to report security vulnerabilities and which versions are supported.

---

## Supported Versions

Only the latest stable release receives security updates.

| Version | Supported |
|--------|-----------|
| v1.0   | ✅ Yes |
| < v1.0 | ❌ No |

---

## Reporting a Vulnerability

If you discover a security vulnerability, **please do not open a public GitHub issue**.

Instead, report it responsibly using the method below.

### Contact
- **Telegram:** `@harsh2hell`  

### What to Include
Please include as much information as possible:
- Description of the vulnerability
- Steps to reproduce the issue
- Affected files, endpoints, or components
- Potential impact (e.g., data exposure, auth bypass)
- Screenshots, logs, or proof-of-concept (if available)

---

## Response Timeline

We aim to follow this response schedule:
- **Initial acknowledgment:** within 48 hours
- **Status update:** within 5 working days
- **Fix or mitigation:** as soon as reasonably possible

---

## Responsible Disclosure

We kindly ask security researchers to:
- Avoid exploiting vulnerabilities beyond proof-of-concept
- Refrain from public disclosure until a fix is released
- Allow reasonable time for investigation and remediation

Responsible disclosure helps keep everyone safe.

---

## Security Scope

This project currently focuses on securing:
- Staff authentication & session handling
- SQLite database operations
- QR-based payment flow logic
- User input validation and form handling

Planned future improvements:
- Role-based access control (RBAC)
- SaaS-grade authentication system
- Secure password hashing & encryption
- Activity and audit logging

---

## ⚠ Disclaimer

This project is an **early-stage POS system (v1.0)** created for learning and development purposes.

It is **not yet intended for high-risk or production-critical environments**.

Use at your own discretion.
